1. Security Architecture Overview

SIVARR is a FastAPI (Python) backend + Vanilla JS/HTML/CSS frontend hosted on Railway with a PostgreSQL database. The security model operates on four layers:

LayerMechanismWhere
TransportHTTPS enforced by Railway TLS terminationInfrastructure
Authenticationbcrypt passwords + random session tokens (DB-backed)app.py
AuthorizationOrg ownership checked on every mutationapp.py + database.py
Output SafetyDOMPurify on all AI-generated HTML; esc() on user dataapp.js
Core principle Every piece of user-supplied data is treated as untrusted until explicitly validated. Org resources are scoped with AND org_id = %s at the SQL level. Token verification is always constant-time.

2. Authentication System

User Registration & Login — /api/login

A single endpoint handles both actions via req.action = "register" | "login".

Password hashing

All passwords are hashed using bcrypt with a randomly generated salt before storage. Plain-text passwords are never written to disk or database.

# ✓ Correct — always do this
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()

# ✗ Never store plain text
user["password"] = password

Adding a new login provider

If you add OAuth (Google, GitHub, etc.), follow this pattern:

  • Generate or look up a user record by email — never trust the OAuth token's sub-claim without verifying the audience.
  • Issue a Sivarr session token (via create_session_token) after OAuth verification. Do not use the OAuth access token as a Sivarr session.
  • Store the OAuth refresh token encrypted at rest (see Section 15 — Known Remaining Items).

Validation rules enforced

  • Name: 2–80 characters, letters/spaces/hyphens/apostrophes only
  • Password: minimum 8 characters (both registration and reset)
  • Email: validated by Pydantic validator, lowercased, deduplicated
  • Passwords confirmed on registration (confirm_password field)

3. Session Management

How tokens work

On successful login, a 32-byte URL-safe random token is generated via secrets.token_urlsafe(32). It is stored in:

  1. An in-memory dict _session_tokens (fast path)
  2. The PostgreSQL sessions table (persistence across restarts)

The token is returned to the browser and stored in localStorage as sivarr_token. Every subsequent API call includes it in the POST body as "token": "...".

PropertyValue
Token entropy256 bits (32 bytes)
Lifetime30 days (SESSION_TTL_DAYS)
Storage (server)In-memory dict + PostgreSQL sessions table
Storage (client)localStorage (not HttpOnly cookie — see Section 15)
Cleanupcleanup_expired_tokens() runs on every login

Validating a token in a new endpoint

# Pattern used throughout app.py
entry = get_session_from_token(sanitize_text(token, 100))
if not entry:
    raise HTTPException(401, "Session expired. Please sign in again.")
sid = entry["sid"]
Important Always sanitize the token string with sanitize_text(token, 100) before passing it to get_session_from_token(). Never call it with raw user input.

Revoking a session (logout)

delete_session_token(token)   # removes from memory + DB

4. Admin & Lecturer Sessions

Admin and lecturer portals use separate short-lived sessions independent of the user session system.

PropertyValue
Token prefixadm_ (admin), lec_ (lecturer)
Entropy256 bits
Lifetime2 hours (_PRIV_SESSION_TTL)
StorageIn-memory only (_admin_sessions, _lecturer_sessions)
Validation_is_valid_admin_session(token) / _is_valid_lecturer_session(token)

Protecting a new admin endpoint

@app.get("/api/admin/my-new-endpoint")
async def my_endpoint(token: str):
    if not _is_valid_admin_session(token):
        raise HTTPException(401, "Unauthorized")
    # ... your logic here

Protecting a new lecturer endpoint

@app.post("/api/lecturer/my-endpoint")
async def my_endpoint(data: dict):
    verify_lecturer(data.get("token", ""))   # raises 401 if invalid/expired
    # ... your logic here
Never do this Do not recreate HMAC-derived tokens. The old pattern "admin_" + hmac(password)[:16] was permanently valid if leaked. All admin/lecturer checks now use the session functions above.

Session expiry notice

Admin and lecturer sessions expire after 2 hours. The admin dashboard JavaScript should check for 401 responses and redirect to the login page when the session expires.

5. Email Verification Gate

New users receive a verification email on registration. Login is blocked until the email is verified.

Flow

  1. User registers → verification email sent in background via Resend.
  2. User tries to log in with unverified email → backend sends a fresh verification link automatically and returns 403 "email_not_verified".
  3. Frontend shows: "Your email isn't verified yet. A new link has been sent — check your inbox."
  4. User clicks the link → /api/auth/verify-email/{token} → marks email verified in DB → redirects to /?verified=1.
  5. User logs in successfully.

Adding a new email-sending action

Use the send_email(to, subject, html_body) helper. Always wrap it in bg.add_task() so it doesn't block the HTTP response.

@app.post("/api/some-action")
async def some_action(data: dict, bg: BackgroundTasks):
    # ... logic ...
    bg.add_task(send_email, user_email, "Subject", html_body)
    return {"ok": True}

DB functions

  • db.create_email_verify_token(sid, email) — create a one-time token
  • db.mark_email_verified(sid) — mark verified
  • db.is_email_verified(sid) — returns bool; always check db.is_available() first
Note — DB unavailability The verification gate is only enforced when the database is reachable. If db.is_available() returns False, login proceeds without the check. This prevents a DB outage from locking all users out.

6. Account Lockout

After 10 consecutive failed password attempts for a given email address, the account is locked for 15 minutes.

SettingVariableDefault
Max attempts before lockLOGIN_LOCK_ATTEMPTS10
Lock durationLOGIN_LOCK_MINUTES15 minutes
Storage_failed_logins dictIn-memory
Restart caveat Because _failed_logins is in-memory, a Railway redeploy resets all counters. If persistence across restarts is required, migrate the failed-login counter to the PostgreSQL users table with a failed_attempts column and a locked_until timestamp.

How it interacts with rate limiting

Two independent controls protect the login endpoint:

  • IP rate limit (check_rate_limit) — limits requests per IP regardless of account
  • Account lockout (_check_account_lockout) — limits attempts per email regardless of IP

Both must pass before the password is checked. This blocks both single-IP brute force and distributed attacks across many IPs targeting one account.

7. Authorization & Ownership Checks

Every endpoint that reads or mutates org data follows this pattern:

sid, _ = _resolve_token(data)                   # 1. authenticate
org = db.get_org_by_member(sid)                 # 2. look up caller's org
if not org: raise HTTPException(404, "No org") # 3. confirm membership

# 4. pass org_id into the DB query — enforced at SQL level
db.update_org_task(task_id, updates, org["id"])

SQL ownership enforcement

All org mutation functions in database.py include AND org_id = %s in the WHERE clause so that even if a bug slips through the application layer, the database rejects cross-org modifications:

UPDATE org_tasks SET ... WHERE id = %s AND org_id = %s
DELETE FROM org_docs  WHERE id = %s AND org_id = %s

Adding a new org resource endpoint

Use this template — do not skip any step:

@app.post("/api/org/myresource/update")
async def my_resource_update(data: dict):
    sid, _ = _resolve_token(data)                        # auth
    if not db.is_available():
        raise HTTPException(503, "Database unavailable.")
    org = db.get_org_by_member(sid)                      # ownership
    if not org:
        raise HTTPException(404, "No organization found.")
    resource_id = sanitize_text(str(data.get("id", "")), 40)
    if not resource_id:
        raise HTTPException(400, "id required.")
    db.update_my_resource(resource_id, updates, org["id"]) # scoped query
    return {"ok": True}

Role-based actions (owner/admin only)

org = db.get_org_by_member(sid)
role = org.get("member_role", "member")   # from DB, not from client
if role not in ("owner", "admin"):
    raise HTTPException(403, "Admins only.")
Never do this Do not read org_id or org_role from the user's progress file (load_progress(sid)). That file is user-writable. Always derive org identity from db.get_org_by_member(sid).

8. Input Validation & Sanitization

sanitize_text(value, max_len)

All text inputs from the client must pass through sanitize_text() before use. It strips null bytes, control characters, and truncates to the specified max length.

title   = sanitize_text(str(data.get("title", "")), 200)
content = sanitize_text(str(data.get("content", "")), 50000)

SQL injection prevention

All database queries use parameterized statements with psycopg2. Never build SQL strings by concatenating user input.

# ✓ Safe — parameterized
cur.execute("SELECT * FROM users WHERE email = %s", (email,))

# ✗ Never do this
cur.execute(f"SELECT * FROM users WHERE email = '{email}'")

Dynamic SET clauses (update functions)

When building dynamic UPDATE queries (e.g. update_org_task), column names come from an explicit allowlist, never from user input directly:

allowed = {"title", "description", "status", "priority"}
sets = {k: v for k, v in data.items() if k in allowed}
cols = ", ".join(f"{k}=%s" for k in sets)   # column name is our constant
cur.execute(f"UPDATE ... SET {cols} WHERE id=%s AND org_id=%s", ...)

Pydantic models for structured requests

Request bodies that use Pydantic BaseModel are validated automatically. For endpoints that accept a raw dict, manually sanitize every field you use.

9. XSS Protection

Backend — template rendering

All user-facing HTML email templates use f-strings with direct variable insertion. Any field that comes from user input must be passed through Python's html.escape() before inserting into email HTML.

Frontend — esc() function

The esc(str) helper in app.js HTML-escapes all user-supplied data before inserting it via innerHTML:

// ✓ Safe
el.innerHTML = `<div>${esc(user.name)}</div>`;

// ✗ Never for user data
el.innerHTML = `<div>${user.name}</div>`;

AI-generated content — DOMPurify

AI responses (Gemini) are passed through renderMarkdown() which produces HTML. The output is then sanitized by DOMPurify before assignment to innerHTML:

function renderMarkdown(text) {
  let h = text
    .replace(/```...```/g, ...)       // code blocks — esc() applied
    .replace(/`([^`]+)`/g, (_, c) =>  // inline code — esc() applied
      `<code>${esc(c)}</code>`)
    ...
  return typeof DOMPurify !== 'undefined'
    ? DOMPurify.sanitize(h, { FORCE_BODY: true })  // final safety net
    : h;
}

Rules for new UI features

  • Use textContent by default. Only use innerHTML when rendering rich content.
  • If using innerHTML with any server-originated data, wrap with DOMPurify.sanitize().
  • If using innerHTML with user-typed input, always run through esc().
  • Never use eval(), Function(), or document.write().

10. CORS Policy

CORS is locked to an explicit list of allowed origins. Wildcard * was removed.

_cors_origins = list({o.strip() for o in [
    BASE_URL.rstrip("/"),
    "https://sivarr-repository-production.up.railway.app",
    "https://sivarr.app",
] if o.strip()})

app.add_middleware(CORSMiddleware,
    allow_origins=_cors_origins,
    allow_methods=["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["Authorization", "Content-Type"],
    allow_credentials=False,
)

Adding a new domain

If SIVARR gets a custom domain or a staging environment, add it to _cors_origins in app.py. Never add "*".

Local development For local testing add "http://localhost:8000" to _cors_origins during development, then remove it before deploying to production.

12. Paystack Payment Security

Webhook verification

The Paystack webhook at /api/webhooks/paystack verifies every incoming request using HMAC-SHA512 with the PAYSTACK_WEBHOOK_SECRET. Never process a webhook event without this check.

Secret key storage

Org-level Paystack keys are stored in the org_integrations table. The secret key is:

  • Validated against the live Paystack API before being saved
  • Never returned to the browser in any response
  • Accessed only by server-side proxy endpoints

Payment verification rule

When verifying a Paystack transaction, only the authenticated session's SID is used as the buyer identity. The metadata.buyer_sid field (which comes from the client) is ignored:

# ✗ Old — trusted client-controlled metadata
buyer_sid = meta.get("buyer_sid", "") or sid or ""

# ✓ New — only the authenticated session
buyer_sid = sid or ""

Adding a new payment flow

  • Always initialize payments server-side and embed buyer_sid = sid (from the authenticated session) in metadata.
  • Always verify the transaction status with the Paystack API server-side before granting access.
  • Use idempotency: check db.check_payment_reference(reference) before processing to prevent double-grants.

13. File Upload Security

The current upload endpoint (/api/upload) accepts .txt, .pdf, and .md files. Validation is extension-based.

Known gap MIME type and magic-byte validation is not yet implemented. A file can be renamed to bypass the extension check. For the current code paths (text decode + pypdf parse), the practical risk is low, but this should be hardened if uploads are ever served back to other users.

Rules for any new upload endpoint

  • Validate both the file extension and the Content-Type header.
  • Read the first 8 bytes of the file and check magic bytes (e.g. PDF starts with %PDF-).
  • Never execute or serve uploaded files from the same origin without content-type sniffing disabled.
  • Store uploaded files outside the web root, or use Railway volumes / cloud object storage.
  • Enforce a file size limit before reading content into memory.
# Example magic-byte check for PDF
content = await file.read(8)
if not content.startswith(b"%PDF-"):
    raise HTTPException(400, "Invalid PDF file.")

14. Password Policies

RuleRegistrationPassword Reset
Minimum length8 characters8 characters
Confirmation requiredYesNo
Algorithmbcryptbcrypt
Reset token expiry—DB-controlled

Reset flow

  1. POST /api/auth/forgot-password — sends a reset link to the email on file.
  2. User clicks link → POST /api/auth/reset-password with token + new password.
  3. Token is single-use: db.mark_reset_token_used(token) is called immediately after use.
Future hardening Consider adding a complexity requirement (at least one number + one symbol). The current 8-character minimum is a baseline. You can add a _validate_password_strength(password) helper and call it from both the registration and reset endpoints.

15. Known Remaining Items

These issues are documented and understood. They are not actively exploitable under current usage patterns but should be addressed as the platform grows.

IDIssueSeverityRecommended Fix
R-1 Session token in localStorage (not HttpOnly cookie). Vulnerable if XSS is introduced. Medium Migrate to HttpOnly; Secure; SameSite=Strict cookie. Requires backend cookie-setting and frontend removal of localStorage token reads.
R-2 GitHub OAuth token stored unencrypted in user progress JSON. Medium Encrypt with cryptography.fernet before writing to DB. Decrypt only when making API calls.
R-3 Google Calendar OAuth uses session bearer token as the OAuth state parameter, leaking it into server logs. Medium Generate a short-lived opaque CSRF token for state; look up the session at callback time using the CSRF token.
R-4 Account lockout counter is in-memory only — resets on Railway redeploy. Low Persist to DB: add failed_attempts INT DEFAULT 0 and locked_until TIMESTAMPTZ columns to users table.
R-5 File uploads validated by extension only, not magic bytes. Low Add magic-byte check as shown in Section 13.
R-6 Deterministic exam question shuffle (seed from known values — student can predict order). Low Generate a random seed at exam-start time; store in the exam session record.
R-7 Sentry DSN exposed in page config response to unauthenticated requests. Low Remove DSN from SIVARR_CONFIG JavaScript injection. Sentry is already initialized via Loader Script in <head>.

16. Security Checklist for Future Features

Run through this checklist every time you add a new API endpoint or frontend feature.

New backend endpoint

  • Validate session token with get_session_from_token() or _resolve_token(data)
  • If org-scoped: call db.get_org_by_member(sid) and pass org["id"] to all DB calls
  • Sanitize every user-supplied string with sanitize_text(value, max_len)
  • All DB queries use parameterized %s placeholders — no f-string SQL
  • Dynamic column names come from an explicit allowlist, not from user input
  • Rate-limit with check_rate_limit(key, limit, "endpoint_name")
  • Admin-only actions check _is_valid_admin_session(token)
  • Org admin-only actions check role from DB: org.get("member_role")
  • Payment actions use authenticated sid, not metadata from client
  • File uploads: validate extension + Content-Type + magic bytes
  • Emails sent via bg.add_task(send_email, ...), never blocking

New frontend feature

  • User-supplied strings inserted via esc(value) before innerHTML
  • AI-generated HTML run through renderMarkdown() which calls DOMPurify
  • No eval(), Function(), or document.write() usage
  • Sensitive tokens not logged to the console
  • API errors handled gracefully — error messages shown via err.textContent, not innerHTML
  • New fetch calls to /api/progress or /api/wrong include &token=... param

New payment integration

  • Webhook signatures verified with HMAC before processing
  • Secret keys stored in org_integrations table — never returned to browser
  • Buyer identity taken from authenticated session, not client metadata
  • Idempotency check before granting access (reference de-duplication)

New OAuth integration

  • OAuth state parameter is an opaque CSRF token — not the Sivarr session token
  • Access tokens stored encrypted at rest
  • Sivarr session issued separately after OAuth verification completes
